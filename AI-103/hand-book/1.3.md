# AI-103 Cheat Sheet — Module 3: Develop a Generative AI Chat App with Microsoft Foundry

## Module Name

**Develop a generative AI chat app with Microsoft Foundry**

---

## 1. Exam Domain Mapping

| AI-103 Exam Domain | Relevance | Why |
|---|---:|---|
| **Implement generative AI and agentic solutions** | **High** | Covers building a chat app, selecting endpoint/SDK/API, generating responses, managing conversation context, and streaming. |

**Exam focus:**  
This module is mainly about how client applications connect to Microsoft Foundry models and how to choose between:

- Project endpoint vs Azure OpenAI endpoint
- Foundry SDK vs OpenAI SDK
- Responses API vs ChatCompletions API
- Key-based auth vs Microsoft Entra ID
- Non-streaming vs streaming/asynchronous response handling

---

## 2. What This Module Teaches

This module teaches how to build a generative AI chat application that connects to a model deployed in a Microsoft Foundry project.

The key message is:

> Building a chat app is not only about sending a prompt to a model. You must choose the correct endpoint, SDK, authentication approach, API style, conversation-state strategy, and response-handling pattern.

The module starts in the Foundry playground, then moves into code using the OpenAI SDK, Responses API, ChatCompletions API, conversation tracking, streaming, and asynchronous clients.

## 3. Key Concepts

### 3.1 Playground-first development

Before writing code, use the **Model playground** to test model behaviour.

| Playground Capability | Why It Matters |
|---|---|
| Send prompts to deployed models | Quickly test model responses. |
| Adjust temperature and max tokens | Tune creativity and output length. |
| Add system messages | Control tone, role and behaviour. |
| Compare models/configurations | Validate model choice before coding. |
| Generate code samples | Start application development from working examples. |

**Exam memory:**  
The playground is for rapid testing and prompt experimentation before application integration.

---

### 3.2 Playground-to-code workflow

```text
Explore in playground
        ↓
Test prompts and model settings
        ↓
Generate code sample
        ↓
Customize application code
        ↓
Iterate between playground and code
```

This is the recommended development flow when building early generative AI applications.

---

### 3.3 Endpoint choices

Each Microsoft Foundry project exposes two important endpoints for model consumption.

| Endpoint | Format / Purpose | Use When |
|---|---|---|
| **Project endpoint** | `https://{resource-name}.services.ai.azure.com/api/projects/{project-name}` | You need Foundry-native project features such as agents, tracing, datasets, indexes, connections or project configuration. |
| **Azure OpenAI endpoint** | `https://{resource-name}.openai.azure.com/openai/v1` | You need broad OpenAI API compatibility for inference with OpenAI SDK patterns. |

**Exam memory:**  
For broad OpenAI API support with Foundry models, use the **Azure OpenAI endpoint**.

---

### 3.4 SDK choices

| SDK | Best For | Key Point |
|---|---|---|
| **Microsoft Foundry SDK / Azure AI Projects** | Project-level features: agents, evaluations, tracing, datasets, indexes, connections, governance. | Use when you need Foundry-native capabilities. |
| **OpenAI SDK** | Model inference, Responses API, ChatCompletions API, image generation, OpenAI compatibility. | Use when you want portable OpenAI-compatible inference code. |

**Python package for Foundry SDK:**  
`azure-ai-projects`

**Exam memory:**  
If the question asks which Python package is used for the Microsoft Foundry SDK, choose **azure-ai-projects**.

---

### 3.5 Authentication choices

| Authentication Method | Use Case | Exam Note |
|---|---|---|
| **Microsoft Entra ID** | Production applications and identity-based access. | Recommended wherever possible. |
| **API key** | Simpler development/testing or supported scenarios. | Store securely, preferably in Azure Key Vault. Never hard-code. |
| **Environment variables** | Local development and configuration. | Useful, but secrets still need secure handling. |

**Production rule:**  
Use **Microsoft Entra ID authentication** where possible. Avoid hard-coded API keys.

---

### 3.6 Responses API

The **Responses API** is the recommended approach for most new Microsoft Foundry chat development.

| Feature | Meaning |
|---|---|
| **Stateful conversations** | Can maintain context using `previous_response_id`. |
| **Unified API** | Combines patterns from ChatCompletions and Assistants-style workflows. |
| **Foundry direct model support** | Works with models hosted directly in Microsoft Foundry. |
| **Simple syntax** | Uses `instructions` and `input`. |
| **Streaming support** | Allows incremental response display. |
| **Async support** | Supports non-blocking application behaviour. |

**Core method:**  
`client.responses.create()`

**Exam memory:**  
If the question asks which method generates responses with the Responses API, choose **client.responses.create()**.

---

### 3.7 ChatCompletions API

The **ChatCompletions API** is the older, widely adopted API pattern.

| Feature | Meaning |
|---|---|
| **Message array** | Uses JSON messages with roles such as `system`, `user`, and `assistant`. |
| **Manual context tracking** | You must send prior messages each turn to retain context. |
| **Cross-platform familiarity** | Common across many models and platforms. |
| **Useful for legacy code** | Often needed for maintenance and compatibility. |

**Core method:**  
`client.chat.completions.create()`

**Exam memory:**  
Responses API is recommended for new projects, but ChatCompletions remains useful for existing code and OpenAI-compatible patterns.

---

### 3.8 Conversation context

There are two main ways to maintain chat context.

| Approach | Used By | How It Works |
|---|---|---|
| **Previous response ID** | Responses API | Pass `previous_response_id` into the next request. |
| **Manual message history** | ChatCompletions API | Store and resend the full conversation message list. |

**Exam trap:**  
The Responses API can track state using response IDs, but token usage can still increase because active context must be considered.

---

### 3.9 Streaming and async

| Pattern | Use When | Benefit |
|---|---|---|
| **Streaming** | Long responses or interactive chat UI. | User sees output incrementally instead of waiting. |
| **Async API** | Long-running requests or concurrent requests. | Non-blocking application behaviour. |
| **Async streaming** | Responsive UI plus async execution. | Best for scalable interactive apps. |

## 5. Decision Rules

| Scenario | Best Choice | Why |
|---|---|---|
| Need Foundry agents, tracing, evaluations, datasets or connections | **Foundry SDK + Project endpoint** | Gives access to project-level Foundry capabilities. |
| Need simple OpenAI-compatible model inference | **OpenAI SDK + Azure OpenAI endpoint** | Maximum OpenAI API compatibility. |
| Building a new chat app | **Responses API** | Recommended for most new development. |
| Maintaining existing OpenAI-style chat code | **ChatCompletions API** | Well-established and widely supported. |
| Need conversation state without manually storing full messages | **Responses API with `previous_response_id`** | Simplifies multi-turn context. |
| Need full control over included conversation history | **Manual message history** | Useful for pruning, persistence and custom context control. |
| Need responsive UI for long answers | **Streaming** | Displays partial output as it arrives. |
| Need non-blocking or concurrent requests | **AsyncOpenAI** | Improves responsiveness/scalability. |
| Production authentication | **Microsoft Entra ID** | Identity-based, more secure than static keys. |
| Must use API key | **Store in Key Vault / environment variables** | Avoid hard-coding secrets. |
| Need no-code prompt testing | **Model playground** | Validate behaviour before coding. |
| Need generated starter code | **Playground Code button** | Produces endpoint/model/settings-based code samples. |

## 6. Architecture Pattern

### 6.1 Chat application architecture

```text
User
  ↓
Chat UI / CLI App
  ↓
Application Code
  ↓
OpenAI-compatible Client
  ↓
Azure OpenAI Endpoint or Project Endpoint
  ↓
Model Deployment
  ↓
Generated Response
  ↓
Conversation State + Streaming + Monitoring
```

---

### 6.2 Endpoint and SDK architecture

```text
Option A — Foundry-native project features

Client App
  ↓
Foundry SDK / AIProjectClient
  ↓
Project Endpoint
  ↓
Project Config | Connections | Tracing | Datasets | Indexes | Agents
  ↓
OpenAI-compatible Chat Client
  ↓
Model Deployment
```

```text
Option B — OpenAI-compatible inference

Client App
  ↓
OpenAI SDK
  ↓
Azure OpenAI Endpoint
  ↓
Model Deployment
  ↓
Responses API or ChatCompletions API
```

---

### 6.3 Conversation context pattern

```text
Turn 1:
User prompt → Model → Response ID saved

Turn 2:
User prompt + previous_response_id → Model → New response ID saved

Turn N:
User prompt + previous_response_id → Model → Updated context
```

---

### 6.4 Streaming pattern

```text
User prompt
  ↓
Request with stream=True
  ↓
Model starts generating
  ↓
Delta events arrive incrementally
  ↓
App prints/displays partial output
  ↓
Response completed event saves response ID
```

## 7. Configuration / Implementation Notes

### 7.1 Foundry SDK setup

Install:

```bash
pip install azure-ai-projects azure-identity openai
```

Common imports:

```python
from azure.identity import DefaultAzureCredential
from azure.ai.projects import AIProjectClient
```

Project endpoint format:

```text
https://{resource-name}.services.ai.azure.com/api/projects/{project-name}
```

Use `AIProjectClient` for:

- resource connections
- project configuration
- tracing
- datasets
- indexes
- Foundry-native project operations

---

### 7.2 OpenAI SDK setup

Install:

```bash
pip install openai azure-identity
```

Azure OpenAI endpoint format:

```text
https://{resource-name}.openai.azure.com/openai/v1
```

Use OpenAI SDK for:

- Responses API
- ChatCompletions API
- image generation
- Foundry direct model inference
- OpenAI-compatible inference workloads

---

### 7.3 Entra ID authentication pattern

```python
from openai import OpenAI
from azure.identity import DefaultAzureCredential, get_bearer_token_provider

token_provider = get_bearer_token_provider(
    DefaultAzureCredential(),
    "https://ai.azure.com/.default"
)

openai_client = OpenAI(
    base_url="https://{resource-name}.openai.azure.com/openai/v1/",
    api_key=token_provider
)
```

Local development often requires:

```bash
az login
```

---

### 7.4 API key pattern

```python
import os
from openai import OpenAI

openai_client = OpenAI(
    api_key=os.getenv("AZURE_OPENAI_API_KEY"),
    base_url="https://{resource-name}.openai.azure.com/openai/v1/"
)
```

**Security note:**  
Do not hard-code keys. Store secrets securely, preferably in Azure Key Vault.

---

### 7.5 Responses API basic pattern

```python
response = openai_client.responses.create(
    model="gpt-4.1",
    instructions="You are a helpful AI assistant.",
    input="What is Microsoft Foundry?"
)

print(response.output_text)
```

Important response properties:

| Property | Meaning |
|---|---|
| `output_text` | Generated text. |
| `id` | Response identifier. |
| `status` | Completion status. |
| `usage` | Token usage. |
| `model` | Model used. |

---

### 7.6 Responses API context tracking

```python
last_response_id = None

response = openai_client.responses.create(
    model=model_deployment,
    instructions="You are a helpful AI assistant.",
    input=input_text,
    previous_response_id=last_response_id
)

last_response_id = response.id
```

---

### 7.7 ChatCompletions API basic pattern

```python
completion = openai_client.chat.completions.create(
    model="gpt-4o",
    messages=[
        {"role": "system", "content": "You are a helpful assistant."},
        {"role": "user", "content": "When was Microsoft founded?"}
    ]
)

print(completion.choices[0].message.content)
```

---

### 7.8 ChatCompletions context tracking

```python
conversation_messages = [
    {"role": "system", "content": "You are a helpful assistant."}
]

conversation_messages.append(
    {"role": "user", "content": input_text}
)

completion = openai_client.chat.completions.create(
    model=model_deployment,
    messages=conversation_messages
)

assistant_message = completion.choices[0].message.content

conversation_messages.append(
    {"role": "assistant", "content": assistant_message}
)
```

---

### 7.9 Streaming pattern

```python
stream = openai_client.responses.create(
    model=model_deployment,
    instructions="You are a helpful AI assistant.",
    input=input_text,
    previous_response_id=last_response_id,
    stream=True
)

for event in stream:
    if event.type == "response.output_text.delta":
        print(event.delta, end="")
    elif event.type == "response.completed":
        last_response_id = event.response.id
```

---

### 7.10 Async pattern

```python
from openai import AsyncOpenAI

async_client = AsyncOpenAI(
    base_url=azure_openai_endpoint,
    api_key=token_provider
)

response = await async_client.responses.create(
    model=model_deployment,
    instructions="You are a helpful AI assistant.",
    input=input_text,
    previous_response_id=last_response_id
)
```

Remember to close async credentials/client sessions where required.

## 8. Exam Traps

| Trap | Correct Understanding |
|---|---|
| Project endpoint and Azure OpenAI endpoint are interchangeable. | They are different. Project endpoint is for Foundry project features; Azure OpenAI endpoint is best for broad OpenAI API compatibility. |
| OpenAI SDK gives access to all Foundry project features. | No. Use Foundry SDK for agents, evaluations, tracing, datasets, indexes and connections. |
| Foundry SDK is only for model inference. | No. Its value is project-level Foundry operations. |
| Responses API and ChatCompletions API are the same. | No. Responses API is newer and recommended for most new projects; ChatCompletions is older and message-array based. |
| ChatCompletions automatically remembers conversation context. | No. You must manually track and resend message history. |
| Responses API makes token usage free/cheap because it stores previous responses. | No. Active context still contributes to token usage. |
| System message and instructions are unrelated. | `instructions` in Responses API serves the same behavioural role as a system message. |
| Streaming improves model intelligence. | No. Streaming improves perceived responsiveness by displaying partial output. |
| Async improves answer quality. | No. Async improves non-blocking execution and concurrency. |
| API keys are preferred for production. | No. Entra ID is preferred where possible. |
| It is safe to put API keys in code. | No. Use Key Vault or secure environment configuration. |
| Use ChatCompletions for all new apps. | Responses API is recommended for most new Foundry development. |
| The model parameter is the base model name. | In deployed Azure/Foundry scenarios, it is commonly the deployment name. |
| Endpoint alone is enough to call the model. | You also need authentication and the deployment/model name. |
