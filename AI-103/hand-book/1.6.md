## 2. What This Module Teaches

Generative AI can create highly useful content, but it can also generate harmful, inaccurate, offensive, discriminatory or unsafe output.

This module teaches a practical responsible AI process:

| Stage | Purpose |
|---|---|
| **Map** | Identify and prioritize potential harms. |
| **Measure** | Test and quantify harms using defined criteria. |
| **Mitigate** | Apply controls across model, safety, prompt/grounding and user experience layers. |
| **Manage** | Prepare for compliant, controlled and monitored operation. |

Key principle:

> Responsible generative AI is not a one-time content filter. It is a lifecycle that starts during planning and continues through testing, release, monitoring, incident response and improvement.

## 3. Key Concepts

### 3.1 Responsible Generative AI

Responsible generative AI means designing, testing and operating generative AI systems to reduce harmful output and manage risk.

Potential harms include:

- offensive, pejorative or discriminatory content
- factual inaccuracies
- unsafe or harmful instructions
- illegal or unethical guidance
- privacy or security risks
- misuse or abuse
- over-reliance by users
- lack of transparency
- poor accessibility or exclusion
- harmful content generation in sensitive domains

**Exam memory:**  
Responsible AI must be considered from the start of solution planning, not only after the model is deployed.

---

### 3.2 The Four-Stage Responsible AI Process

```text
Map → Measure → Mitigate → Manage
```

| Stage | What Happens | Exam Tip |
|---|---|---|
| **Map** | Identify, prioritize, test and document potential harms. | Start by understanding risks. |
| **Measure** | Create test prompts, generate outputs and classify harm level. | Establish a baseline. |
| **Mitigate** | Apply layered controls across the solution. | Use defense in depth. |
| **Manage** | Prepare release, operations, incident response and monitoring. | Responsible AI continues in production. |

**Exam memory:**  
This aligns closely with risk management thinking: identify risk, quantify risk, reduce risk, operate responsibly.

---

### 3.3 Map Potential Harms

The mapping stage includes four steps:

```text
Identify → Prioritize → Test / Verify → Document / Share
```

| Step | Meaning |
|---|---|
| **Identify potential harms** | List harms relevant to the specific use case, model, data, audience and deployment context. |
| **Prioritize harms** | Assess likelihood and impact. |
| **Test and verify harms** | Use testing, including red teaming, to verify whether harms occur. |
| **Document and share harms** | Maintain a shared record of known risks and evidence. |

### Examples of Harms

| Harm Type | Example |
|---|---|
| **Offensive content** | Hate, discriminatory or pejorative language. |
| **Factual inaccuracy** | Confident but incorrect advice. |
| **Illegal or unethical support** | Instructions to commit crimes or cause harm. |
| **Unsafe domain guidance** | Dangerous instructions in health, safety, chemicals, cybersecurity or self-harm domains. |
| **Misuse** | Users deliberately probing or abusing the system. |

---

### 3.4 Prioritizing Harms

Prioritization considers:

- intended use
- foreseeable misuse
- likelihood
- impact
- affected users
- severity
- legal or compliance exposure
- organizational risk appetite

Example logic:

| Harm | Likelihood | Impact | Priority |
|---|---:|---:|---:|
| Inaccurate cooking time | High | Medium | Medium–High |
| Poison recipe generation | Low | Very High | High |

**Exam memory:**  
A low-likelihood harm can still be high priority if the impact is severe.

---

### 3.5 Red Teaming

Red teaming is deliberate adversarial testing to find weaknesses and harmful outputs.

For generative AI, red teamers may test:

- jailbreak attempts
- harmful instruction requests
- unsafe domain prompts
- discriminatory content
- misinformation generation
- prompt injection
- policy evasion
- system prompt subversion

**Exam memory:**  
Red teaming helps verify whether mapped harms are realistic and under what conditions they occur.

---

### 3.6 Measure Potential Harms

Measuring harms creates a baseline.

The general measurement process is:

```text
Prepare test prompts → Generate outputs → Evaluate / categorize harm
```

| Step | Meaning |
|---|---|
| **Prepare prompts** | Create diverse test inputs likely to trigger each identified harm. |
| **Submit prompts** | Run prompts through the system. |
| **Evaluate outputs** | Apply predefined criteria to classify outputs. |
| **Document results** | Share evidence and baseline metrics with stakeholders. |

### Manual vs Automated Testing

| Testing Type | Use |
|---|---|
| **Manual testing** | Start with a small set to validate criteria and understand failure modes. |
| **Automated testing** | Scale measurement across many prompts and scenarios. |
| **Periodic manual review** | Ensure automated evaluation still works and captures new risks. |

**Exam memory:**  
Start manually to make sure criteria are valid, then automate larger-scale harm measurement.

---

### 3.7 Mitigate Potential Harms

Mitigation uses a layered approach.

The four mitigation layers are:

```text
Model → Safety System → System Message / Grounding → User Experience
```

| Layer | Mitigation Examples |
|---|---|
| **Model layer** | Select appropriate model, use smaller/specialized model, fine-tune for scope. |
| **Safety system layer** | Foundry guardrails, content filters, prompt shields, abuse detection. |
| **System message and grounding layer** | Prompt engineering, behavioral constraints, RAG from trusted data. |
| **User experience layer** | Constrained UI, input/output validation, warnings, transparency, feedback/reporting. |

**Exam memory:**  
Guardrails are important, but they are only one layer of responsible AI.

---

### 3.8 Model Layer

At the model layer, reduce risk by selecting or tuning the model appropriately.

Examples:

- choose a model fit for the intended use
- avoid overly powerful general models when a simpler model is enough
- fine-tune a model to keep outputs scoped to the scenario
- evaluate model behaviour before release

**Exam memory:**  
Model choice is a responsible AI decision, not only a performance decision.

---

### 3.9 Safety System Layer

Microsoft Foundry provides guardrails and content filtering.

Guardrails can help suppress harmful prompts and responses based on content filters.

Common risk categories include:

- hate and fairness
- sexual content
- violence
- self-harm
- task adherence

Severity levels include:

```text
Safe → Low → Medium → High
```

Other safety-system mitigations include:

- prompt shields
- abuse detection
- filtering for prompts and completions
- threshold-based blocking
- risk-specific controls

**Exam memory:**  
Microsoft Foundry **guardrails** help mitigate harmful content generation at the safety system layer.

---

### 3.10 System Message and Grounding Layer

This layer focuses on prompt construction and context.

Mitigations include:

- system messages that define allowed behaviour
- clear boundaries
- “when unsure” policies
- grounding with trusted data
- RAG to reduce fabricated answers
- prompt engineering for safe behaviour
- instructions to refuse unsafe requests

**Exam memory:**  
Grounding helps reduce factual inaccuracies, but does not replace content filters, validation or monitoring.

---

### 3.11 User Experience Layer

The user experience layer includes the application interface and user-facing documentation.

Mitigations include:

- constrain input options
- limit supported topics
- validate inputs and outputs
- show disclaimers where appropriate
- explain limitations
- provide feedback/report issue controls
- allow content reporting
- avoid overclaiming model capabilities
- design escalation or human review for sensitive scenarios

**Exam memory:**  
Transparency about limitations and risks is part of responsible AI.

---

### 3.12 Manage Responsible AI Solution

Before release, complete reviews and readiness planning.

Common prerelease reviews:

- legal
- privacy
- security
- accessibility
- compliance
- risk
- responsible AI review

Operational planning should include:

- phased rollout
- incident response plan
- rollback plan
- harmful-response blocking capability
- misuse blocking for users/apps/IP addresses
- feedback and reporting channels
- telemetry and monitoring
- privacy-compliant logging

**Exam memory:**  
A phased rollout allows the team to gather feedback and identify issues before a wider release.

## 5. Decision Rules

| Scenario | Best Choice | Why |
|---|---|---|
| Starting responsible AI planning | **Map potential harms** | Understand relevant risks first. |
| Need to document purpose, expected use and harms | **AI Impact Assessment** | Records responsible AI design assumptions. |
| Need to rank risks | **Prioritize by likelihood and impact** | Focus on most important harms. |
| Need to verify whether harms occur | **Red teaming / adversarial testing** | Probes for weaknesses and unsafe responses. |
| Need harm baseline | **Measure outputs using predefined criteria** | Quantifies current risk. |
| Need scalable harm testing | **Automated evaluation plus periodic manual review** | Scales testing while keeping human validation. |
| Harmful content needs blocking | **Foundry guardrails / content filters** | Safety-system level mitigation. |
| User attempts prompt injection or abuse | **Prompt shields / abuse detection** | Detects systematic abuse or subversion attempts. |
| Model lacks factual grounding | **RAG with trusted sources** | Reduces fabricated responses. |
| Model generates out-of-scope content | **System message + UI constraints + filters** | Multiple layers reduce risk. |
| Need to reduce risk at model layer | **Choose appropriate model or fine-tune** | Controls risk at model foundation. |
| Need safe release | **Phased delivery plan** | Limits exposure while gathering feedback. |
| Need production readiness | **Incident response + rollback plan** | Prepares for harmful or unexpected outcomes. |
| User reports harmful output | **Feedback/reporting workflow + blocking capability** | Supports rapid operational response. |
| High-impact actions | **Human review / approval gates** | Reduces automation risk. |

## 7. Configuration / Implementation Notes

### 7.1 Mapping Harms Checklist

Use this checklist:

- Identify intended users.
- Identify intended use.
- Identify foreseeable misuse.
- Review model/service documentation and transparency notes.
- Review industry/compliance requirements.
- Identify offensive, inaccurate, unsafe and illegal output risks.
- Prioritize by likelihood and impact.
- Red-team high-priority risks.
- Document and share verified harms.

---

### 7.2 Measuring Harms Checklist

To measure harms:

1. Prepare diverse test prompts for each harm.
2. Run prompts through the system.
3. Capture generated output.
4. Apply strict evaluation criteria.
5. Categorize harm level.
6. Create a baseline.
7. Document results.
8. Repeat after mitigations.
9. Use automated testing for scale.
10. Keep periodic manual validation.

---

### 7.3 Mitigation Layer Checklist

| Layer | Checklist |
|---|---|
| **Model** | Select appropriate model; avoid unnecessary capability; consider fine-tuning; evaluate model behaviour. |
| **Safety System** | Configure guardrails; content filters; thresholds; prompt shields; abuse detection. |
| **System Message / Grounding** | Add role, boundaries, safe completion rules, RAG, trusted context. |
| **User Experience** | Constrain inputs, validate outputs, show limitations, collect feedback, enable reporting. |

---

### 7.4 Foundry Guardrails

Guardrails can be configured in Microsoft Foundry to control risks.

Common steps:

1. Open the Foundry portal.
2. Select **Guardrails**.
3. Create a new guardrail.
4. Add risk controls.
5. Select categories such as Hate, Violence, Sexual and Self-harm.
6. Set blocking thresholds.
7. Apply guardrail to model deployment.
8. Review and submit.
9. Confirm the guardrail is applied on the model details page.
10. Retest prompts in the playground.

---

### 7.5 Content Filtering Categories

| Category | Purpose |
|---|---|
| **Hate and fairness** | Reduce hate, bias, discriminatory or unfair content. |
| **Sexual** | Reduce inappropriate sexual content. |
| **Violence** | Reduce violent or harmful content. |
| **Self-harm** | Reduce harmful self-injury content. |
| **Task adherence** | Helps ensure responses stay aligned to task requirements. |

Blocking thresholds determine when content is intercepted and prevented.

---

### 7.6 Default vs Custom Guardrails

| Guardrail Type | Meaning | Use When |
|---|---|---|
| **Default guardrail** | Balanced default filtering applied to model deployment. | General-purpose starting point. |
| **Custom guardrail** | Scenario-specific content filter and risk threshold configuration. | Default controls are not enough or policy requires stricter controls. |

**Exam memory:**  
Custom guardrails let you take greater control over harmful or offensive content generation.

---

### 7.7 Release Readiness Checklist

Before release, confirm:

- harms are mapped and prioritized
- baseline measurement exists
- mitigations are applied
- guardrails are configured
- red team findings are reviewed
- legal/privacy/security/accessibility reviews are complete
- user documentation explains capabilities and limitations
- phased rollout plan exists
- incident response plan exists
- rollback plan exists
- telemetry and feedback mechanisms exist
- harmful response blocking capability exists
- user/application/IP blocking capability exists for misuse

## 8. Exam Traps

| Trap | Correct Understanding |
|---|---|
| Responsible AI starts after deployment. | No. It starts during planning and continues through operation. |
| Content filters alone make a solution responsible. | No. Use a lifecycle and layered mitigations. |
| Guardrails are applied at the model layer. | Guardrails are a safety-system layer mitigation. |
| RAG is a safety system. | RAG belongs mainly to grounding/prompt layer and helps reduce unsupported factual output. |
| A system message guarantees safe behaviour. | No. It guides behaviour but must be combined with other controls. |
| Red teaming is only for cybersecurity. | No. It also helps test harmful generative AI behaviours. |
| Measure harms only after mitigation. | No. Measure first to create a baseline, then compare improvements. |
| Automated testing removes the need for manual review. | No. Periodic manual validation is still needed. |
| Phased rollout eliminates the need for risk planning. | No. It reduces exposure while feedback and issues are gathered. |
| Custom guardrails are always unnecessary because default guardrails exist. | No. Custom guardrails are useful for scenario-specific risk thresholds. |
| User feedback is optional in responsible operation. | Feedback/reporting is important for identifying problems in production. |
| Telemetry can be collected without privacy considerations. | Telemetry must comply with privacy laws and organizational commitments. |
| Harm likelihood is the only prioritization factor. | Impact also matters; low likelihood/high impact harms may be high priority. |
| Fine-tuning is a complete safety solution. | No. It can scope behaviour but does not replace guardrails and monitoring. |
| Transparency means exposing internal model prompts. | No. It means clearly communicating capabilities, limitations and risks. |
