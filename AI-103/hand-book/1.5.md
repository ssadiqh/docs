## 2. What This Module Teaches

A base model may produce good general answers, but it may not meet application requirements for:

- accuracy
- consistency
- current knowledge
- private/domain knowledge
- tone and style
- output format
- safety and reliability
- cost and latency

This module teaches three complementary optimisation strategies:

| Strategy | Main Purpose |
|---|---|
| **Prompt engineering** | Guide model behaviour, tone, format and task execution. |
| **Retrieval Augmented Generation (RAG)** | Provide factual, current or private knowledge at query time. |
| **Fine-tuning** | Improve consistency of style, format, tone or tool-use behaviour. |

Key principle:

> Start simple with prompt engineering, add RAG when the model lacks knowledge, and use fine-tuning when behaviour or format needs to be consistently learned.

## 3. Key Concepts

### 3.1 Prompt Engineering

Prompt engineering is the process of designing and refining prompts to improve model response quality.

It is the first optimisation step because it is:

- quick
- low-cost
- low-complexity
- easy to iterate
- does not require additional infrastructure
- does not require training data

### Prompt Components

| Component | Purpose |
|---|---|
| **System message** | Defines role, behaviour, tone, constraints and output format. |
| **User message** | The user’s request or task. |
| **Assistant message** | Previous model response used for conversation context. |
| **Examples** | Input/output examples for few-shot learning. |

### System Message

A system message can define:

- assistant role
- boundaries
- tone
- output format
- safety constraints
- what to do when unsure
- out-of-scope behaviour

Example:

```text
You are a friendly travel advisor for Margie's Travel.
Answer only questions related to travel, hotels, and trip planning.
Use a warm, conversational tone.
If you don't have enough information, ask a clarifying question.
Format hotel recommendations as a bulleted list.
```

**Exam memory:**  
A system message guides behaviour, but it does **not guarantee** compliance. You still need testing, evaluation and guardrails.

---

### 3.2 Prompt Patterns

| Pattern | What It Does | Use When |
|---|---|---|
| **Persona pattern** | Makes the model respond from a specific role or perspective. | Need tone, audience or role-specific style. |
| **Format template pattern** | Forces output into a structure. | Need consistent, parseable output. |
| **Chain-of-thought / step-by-step pattern** | Encourages stepwise reasoning for complex tasks. | Need reasoning transparency or task decomposition. |
| **Few-shot learning** | Provides examples of desired inputs and outputs. | Need pattern matching or consistent classification. |
| **Clear delimiters** | Separates instructions, source data and examples. | Need to reduce ambiguity or prompt confusion. |

### Delimiters

Use clear syntax such as:

```text
---
SOURCE TEXT
---

<instructions>
...
</instructions>

### Examples
...
```

**Exam memory:**  
Use delimiters when prompts include multiple sections such as instructions, data and examples.

---

### 3.3 Model Parameters

| Parameter | What It Controls | Exam Tip |
|---|---|---|
| **temperature** | Randomness and creativity. | Lower for factual tasks, higher for creative tasks. |
| **top_p** | Limits output to the most probable token set. | Alternative to temperature. |
| **max tokens / max output tokens** | Maximum length of the model response. | Controls response length and cost. |

**Important rule:**  
Adjust **temperature OR top_p**, not both at the same time.

### Temperature Examples

| Scenario | Recommended Temperature |
|---|---:|
| Factual answer | Low, such as 0.2 |
| Structured extraction | Low |
| Legal/compliance summary | Low |
| Creative itinerary | Higher, such as 0.7 |
| Brainstorming | Higher |

### 3.4 Retrieval Augmented Generation (RAG)

RAG grounds model responses in external data.

It is used when the model needs:

- private organisational data
- current data
- product catalog data
- policy documents
- frequently changing information
- factual grounding
- information beyond the model’s training cutoff

### RAG Pattern

```text
Retrieve → Augment → Generate
```

| Step | Meaning |
|---|---|
| **Retrieve** | Search a trusted data source for relevant content. |
| **Augment** | Add retrieved content to the prompt as grounding context. |
| **Generate** | Model answers using the supplied context. |

**Exam memory:**  
Prompt engineering cannot give the model knowledge it does not have. Use **RAG** when the model needs access to domain-specific, current or private data.

---

### 3.5 Embeddings and Vector Search

An **embedding** is a vector representation of text meaning.

Embeddings help find semantically similar content even when the words are different.

Example:

```text
"The children played joyfully in the park."
"Kids happily ran around the playground."
```

These sentences use different words but have similar meaning, so their embeddings should be close in vector space.

### Similarity

| Concept | Meaning |
|---|---|
| **Vector** | Numeric representation of text meaning. |
| **Cosine similarity** | Measures closeness between vectors. |
| **Value near 1** | Strong semantic similarity. |
| **Embedding model** | Converts text into vectors. |

---

### 3.6 Azure AI Search for RAG

Azure AI Search provides the retrieval layer for RAG.

Typical flow:

```text
Documents / Data Source
        ↓
Chunking + Embeddings
        ↓
Azure AI Search Index
        ↓
User Question
        ↓
Search / Retrieval
        ↓
Grounding Context
        ↓
Model Response
```

### Search Methods

| Search Type | Meaning | Exam Note |
|---|---|---|
| **Keyword search** | Matches exact terms. | Useful but can miss semantic matches. |
| **Semantic search** | Matches meaning using semantic ranking. | Better natural-language relevance. |
| **Vector search** | Uses embeddings to find similar meaning. | Core for RAG. |
| **Hybrid search** | Combines keyword, semantic and vector search. | Recommended for generative AI apps. |

**Exam memory:**  
For generative AI RAG, **hybrid search** is often preferred because it combines multiple retrieval strengths.

---

### 3.7 Fine-tuning

Fine-tuning adapts a pretrained model using task-specific examples.

It is used to improve:

- consistent tone
- consistent format
- consistent style
- reliable output schema
- response behaviour
- tool selection behaviour
- shorter prompts
- model distillation

Fine-tuning updates model behaviour using training examples, rather than adding live external facts at query time.

**Exam memory:**  
Fine-tuning is for **behaviour consistency**.  
RAG is for **factual grounding**.

---

### 3.8 LoRA

Microsoft Foundry fine-tuning uses **LoRA** — Low-Rank Adaptation.

| Concept | Meaning |
|---|---|
| **LoRA** | Updates a smaller subset of model parameters rather than retraining the whole model. |
| **Benefit** | Faster and more cost-effective than full retraining. |
| **Purpose** | Specialise a general model for specific response patterns. |

---

### 3.9 Fine-tuning Types

| Fine-tuning Type | Purpose | Best For |
|---|---|---|
| **Supervised Fine-Tuning (SFT)** | Train on labelled prompt-response examples. | Clear expected output patterns. |
| **Reinforcement Fine-Tuning (RFT)** | Optimise through iterative feedback/rewards. | Complex or dynamic reasoning tasks. |
| **Direct Preference Optimization (DPO)** | Align using preferred vs non-preferred responses. | Human preference alignment. |

---

### 3.10 Fine-tuning Data

Training examples are usually stored in **JSONL** format.

Example structure:

```json
{"messages": [
  {"role": "system", "content": "You are a friendly travel advisor."},
  {"role": "user", "content": "What's a good beach destination in Europe?"},
  {"role": "assistant", "content": "For a beautiful European beach experience, consider the Algarve..."}
]}
```

Training data should be:

- high quality
- representative
- consistent
- aligned to the desired style and format
- large enough to cover expected scenarios
- free from bias and poor examples

**Exam memory:**  
Include a consistent system message in training examples, and use the same system message when deploying the fine-tuned model.

## 5. Decision Rules

| Scenario | Best Choice | Why |
|---|---|---|
| Need quick improvement in tone or format | **Prompt engineering** | Fastest and lowest complexity. |
| Need a specific output structure | **Format template + examples** | Guides response shape. |
| Need consistent classification | **Few-shot examples** | Shows the pattern. |
| Need current or private data | **RAG** | Retrieves facts at query time. |
| Need answers from product catalog, policy docs or internal KB | **RAG with Azure AI Search** | Grounds output in trusted data. |
| Need semantic retrieval | **Vector search / hybrid search** | Finds meaning, not just exact keywords. |
| Need best retrieval for generative AI app | **Hybrid search** | Combines keyword, semantic and vector retrieval. |
| Need brand voice consistency | **Fine-tuning** | Embeds behaviour into model. |
| Need reliable JSON or repeated output style | **Fine-tuning if prompting fails** | Improves consistency. |
| Need lower prompt length | **Fine-tuning** | Reduces need for long system messages/examples. |
| Need factual accuracy | **RAG, not fine-tuning** | Fine-tuning is not for live factual grounding. |
| Need both factual grounding and brand voice | **RAG + fine-tuning + prompt engineering** | Each strategy solves a different problem. |
| Prompt is long but behaviour still inconsistent | **Consider fine-tuning** | Model learns desired pattern. |
| Data changes frequently | **RAG** | Avoid retraining for every data change. |
| Training data is poor or limited | **Avoid fine-tuning** | Bad training data can degrade behaviour. |
| Unsure where to start | **Start with prompt engineering** | Lowest cost and simplest. |

## 6. Architecture Pattern

### 6.1 Optimisation Decision Flow

```text
Base Model Output
        ↓
Is tone/format/behaviour weak?
        ↓
Start with Prompt Engineering
        ↓
Does the model lack facts/private/current data?
        ↓
Add RAG
        ↓
Is behaviour still inconsistent despite prompting?
        ↓
Consider Fine-tuning
        ↓
Evaluate and Iterate
```

---

### 6.2 Prompt Engineering Architecture

```text

   ↓
System Message + User Message + Examples + Parameters
   ↓
Model Deployment
   ↓
Response
   ↓
```

## 7. Configuration / Implementation Notes

### 7.1 Prompt Engineering Checklist

Use this checklist:

- Define assistant role.
- Define boundaries.
- Specify output format.
- Add examples if needed.
- Use delimiters for source text.
- Add “when unsure” behaviour.
- Tune temperature or top_p.
- Evaluate outputs.
- Iterate.

---

### 7.2 System Message Design

A good system message should answer:

| Question | Example |
|---|---|
| Who is the assistant? | “You are a travel advisor…” |
| What should it do? | “Help users plan trips…” |
| What should it avoid? | “Do not recommend hotels or flights…” |
| What tone should it use? | “Warm and conversational…” |
| What format should it follow? | “Use bullets with name, location, price range…” |
| What if unsure? | “Ask a clarifying question.” |

---

### 7.3 Prompt Parameter Configuration

| Parameter | Low Value | High Value |
|---|---|---|
| **temperature** | More deterministic/factual | More creative/varied |
| **top_p** | Restricts candidate tokens | Allows broader candidate tokens |
| **max output tokens** | Shorter response/lower cost | Longer response/higher cost |

**Exam rule:**  
For factual or compliance-sensitive tasks, use lower temperature.

---

### 7.4 RAG Implementation Steps

Typical RAG implementation:

1. Prepare source documents.
2. Chunk the documents.
3. Create embeddings.
4. Store chunks and embeddings in Azure AI Search.
5. Convert user question to an embedding.
6. Retrieve relevant chunks.
7. Add retrieved context to the prompt.
8. Generate grounded response.
9. Evaluate groundedness and relevance.

---

### 7.5 Azure AI Search Data Sources

Possible data sources include:

- Azure Blob Storage
- Azure Data Lake Storage Gen2
- Microsoft OneLake
- uploaded files
- enterprise data sources, depending on architecture

---

### 7.6 RAG Code Pattern

```python
from azure.ai.projects import AIProjectClient
from azure.identity import DefaultAzureCredential
import os

project = AIProjectClient(
    endpoint=os.environ["PROJECT_ENDPOINT"],
    credential=DefaultAzureCredential(),
)

client = project.get_openai_client()

response = client.responses.create(
    model="gpt-4o",
    input=[
        {
            "role": "system",
            "content": "You are a helpful travel advisor. "
                       "Use the following hotel data to answer: " + retrieved_context
        },
        {
            "role": "user",
            "content": "Which hotels do you offer in Paris?"
        }
    ],
)

print(response.output_text)
```

**Exam note:**  
`retrieved_context` represents content returned from the search index.

### 7.7 Fine-tuning Workflow

Typical Foundry fine-tuning workflow:

1. Deploy a base model.
2. Test baseline behaviour.
3. Prepare JSONL training data.
4. Start fine-tuning job.
5. Select base model, customization method and training data.
6. Configure deployment option.
7. Wait for job completion.
8. Deploy fine-tuned model.
9. Compare base model vs fine-tuned model.
10. Evaluate consistency, tone, format and safety.

---

### 7.8 Fine-tuning Training Data Checklist

Training examples should:

- use JSONL format
- include system, user and assistant messages
- include a consistent system message
- reflect the exact desired tone and style
- cover expected user scenarios
- avoid poor-quality responses
- include enough examples to generalise
- be reviewed for bias and safety issues

---

### 7.9 Fine-tuning Risks

| Risk | Meaning |
|---|---|
| **Training cost** | Upfront compute cost. |
| **Hosting cost** | Ongoing cost to host custom model. |
| **Poor data quality** | Bad examples create bad behaviour. |
| **Overfitting** | Model becomes too narrow. |
| **Underfitting** | Model fails to learn desired pattern. |
| **Maintenance** | Retraining may be needed as requirements/base models change. |
| **Drift** | Fine-tuned model may become less effective over time. |

## 8. Exam Traps

| Trap | Correct Understanding |
|---|---|
| Prompt engineering permanently changes the model. | No. It only influences responses at runtime. |
| A system message guarantees compliance. | No. It guides behaviour but needs testing and guardrails. |
| RAG changes model weights. | No. RAG retrieves context at query time. |
| Fine-tuning is best for adding factual knowledge. | No. Use RAG for factual/current/private knowledge. |
| RAG is best for consistent tone. | No. RAG improves grounding; fine-tuning improves consistency. |
| Fine-tuning should be the first optimisation step. | No. Start with prompt engineering. |
| Temperature controls response length. | No. It controls randomness/creativity. |
| max output tokens controls creativity. | No. It controls maximum response length. |
| You should always adjust temperature and top_p together. | No. Usually adjust one or the other. |
| Vector search requires exact keyword match. | No. It uses embeddings to match semantic meaning. |
| Keyword search is always enough for RAG. | No. Hybrid search is generally better for generative AI apps. |
| Fine-tuning removes the need for evaluation. | No. It requires baseline comparison and ongoing evaluation. |
| Training data quantity matters more than quality. | No. High-quality representative examples matter most. |
| A fine-tuned model never needs retraining. | No. It may need retraining when requirements or base models change. |
| Prompt engineering, RAG and fine-tuning are competing options. | No. They are complementary and can be combined. |
