**Exam focus:**  
This module is mainly about how tools extend a generative AI model beyond basic chat by allowing it to retrieve information, run code, search files, search the web, or call application functions.

---

## 2. What This Module Teaches

This module teaches how to build generative AI applications that use tools through the **OpenAI Responses API**.

The key idea is:

> A model without tools can reason and generate text. A model with tools can retrieve current information, ground responses in files, run code, call business functions, and support more practical workflows.

The module introduces four important tool types:

| Tool | Purpose |
|---|---|
| **`code_interpreter`** | Lets the model generate and run Python code in a sandboxed runtime. |
| **`web_search`** | Lets the model retrieve current public web information. |
| **`file_search`** | Lets the model search uploaded files indexed in a vector store. |
| **`function`** | Lets the model request custom functions that your application runs. |

Important distinction:

> Tools in a Responses API request are **not the same as Foundry Tools**. Responses API tools are specified in the prompt request. Foundry Tools are Azure AI APIs such as Language, Speech, Translator, Document Intelligence, and Content Understanding.

## 3. Key Concepts

### 3.1 Why tools matter

Tools help bridge the gap between model reasoning and real-world action.

| Without Tools | With Tools |
|---|---|
| Model relies mainly on training data. | Model can access current or private information. |
| Answers may be generic or outdated. | Answers can be grounded in web results, files, or system data. |
| Model cannot execute code. | Model can run Python through `code_interpreter`. |
| Model cannot directly trigger business logic. | Model can request function calls handled by your application. |
| Limited to text generation. | Can support multi-step workflows and task automation. |

---

### 3.2 Tool configuration pattern

Tools are provided in the `tools` array of a `responses.create()` call.

```python
response = client.responses.create(
    model=model_deployment,
    instructions="You are a helpful AI assistant.",
    input="Find information about vintage computers.",
    tools=[
        {
            "type": "web_search"
        },
        {
            "type": "file_search",
            "vector_store_ids": [vector_store.id]
        }
    ]
)
```

The model decides whether to use a tool unless you configure stronger tool-selection rules or guide the decision through instructions.

---

### 3.3 Tool types

| Tool | What It Does | Best Use Case |
|---|---|---|
| **`code_interpreter`** | Runs Python code in a sandboxed environment. | Calculations, data analysis, file processing, simulations. |
| **`web_search`** | Searches the public web at runtime. | Current events, recent facts, product updates, policies, market information. |
| **`file_search`** | Searches uploaded files indexed in a vector store. | Internal documents, policy Q&A, manuals, contracts, support knowledge. |
| **`function`** | Requests developer-defined function calls. | APIs, databases, workflows, business logic, system integration. |

---

### 3.4 Model decides when to use tools

By default, the model chooses:

- whether a tool is needed
- which tool to use
- what query or function call to issue
- how to incorporate the result into the final answer

Your application still controls:

- which tools are available
- what data sources are connected
- which function calls are allowed
- how function calls are executed
- whether high-impact actions need approval
- how outputs are validated, logged, and governed

---

### 3.5 Tools vs agents

This module focuses on **tools specified by the client application in a model request**.

Agentic AI goes further by persisting:

- model
- instructions
- tools
- memory/knowledge
- orchestration behaviour
- safeguards
- approval workflows

| Pattern | Tool Configuration Lives In | Best For |
|---|---|---|
| **Generative AI app with tools** | Client application request | Custom app-controlled assistant logic. |
| **Foundry agent** | Persisted agent configuration | Reusable, managed, agentic workflows. |

## 5. Decision Rules

| Scenario | Best Choice | Why |
|---|---|---|
| Need calculations, statistics, data transformation, or file conversion | **`code_interpreter`** | Lets the model generate and run Python code. |
| Need current information from the internet | **`web_search`** | Retrieves fresh public web information at runtime. |
| Need answers from uploaded private documents | **`file_search`** | Searches indexed file content in vector stores. |
| Need to call internal APIs, databases, or business workflows | **`function` tool** | Your app executes controlled business logic. |
| Need enterprise-scale knowledge across many stores | **Foundry IQ with agents** | More scalable than simple file search for large knowledge integration. |
| Need real-time current facts | **`web_search`** | Model training data may be outdated. |
| Need authoritative internal policy answers | **`file_search`** | Ground answers in controlled source documents. |
| Need high-impact actions such as submitting orders or sending emails | **Function tool with approval/authorization** | Application must control and validate actions. |
| Need deterministic business logic | **Function tool** | Do not rely on the model to infer business rules. |
| Need model to choose between available tools | **Provide multiple tools in `tools` array** | Model can select based on prompt. |
| Need safer production behaviour | **Validate inputs, outputs, tool calls, and permissions** | Tool outputs can affect real systems. |
| Need production authentication | **Microsoft Entra ID** | Preferred over API keys where possible. |
| Need to use API keys | **Store securely in Key Vault/environment variables** | Never hard-code secrets. |

## 6. Architecture Pattern

### 6.1 Tool-enabled generative AI app

```text
User
  ↓
Client Application
  ↓
Responses API Request
  ├── model
  ├── instructions
  ├── input
  └── tools[]
        ├── code_interpreter
        ├── web_search
        ├── file_search
        └── function
  ↓
Model decides whether a tool is needed
  ↓
Tool execution / retrieval / function call
  ↓
Tool result returned to model
  ↓
Final grounded response
  ↓
Application displays answer / takes next action
```

---

### 6.2 `code_interpreter` pattern

```text
User asks calculation / analysis question
  ↓
Model decides Python is useful
  ↓
Model generates Python code
  ↓
Code runs in sandboxed environment
  ↓
Execution result returned to model
  ↓
Model explains final result
```

---

### 6.3 `web_search` pattern

```text
User asks time-sensitive question
  ↓
Model decides fresh data is needed
  ↓
Web search query is issued
  ↓
Relevant web results are reviewed
  ↓
Model generates source-aware answer
```

---

### 6.4 `file_search` pattern

```text
Source files
  ↓
Upload to vector store
  ↓
Files indexed as searchable chunks
  ↓
User asks document-specific question
  ↓
Model searches vector store
  ↓
Relevant passages are injected into context
  ↓
Model answers from retrieved content
```

---

### 6.5 `function` tool pattern

```text
User asks for system-specific information/action
  ↓
Model emits function_call
  ↓
Application validates call
  ↓
Application runs matching function/API/business logic
  ↓
Application sends function_call_output back to model
  ↓
Model generates final response using tool result
```

**Important:**  
The model does **not** run your business function automatically. Your application must run it and return the result.

## 7. Configuration / Implementation Notes

### 7.1 Basic tool declaration pattern

```python
response = client.responses.create(
    model=model_deployment,
    instructions="You are a helpful AI assistant.",
    input=input_text,
    tools=[
        {"type": "web_search"}
    ]
)
```

---

### 7.2 `code_interpreter` configuration

```python
response = client.responses.create(
    model=model_deployment,
    instructions="Use the python tool to run code for math problems.",
    input="What is the square root of 16?",
    tools=[
        {
            "type": "code_interpreter",
            "container": {"type": "auto"}
        }
    ]
)
```

Key points:

- Runs Python in a sandbox.
- Useful for math, data analysis and file processing.
- Can use common libraries such as `pandas`, `numpy`, `math`, and `matplotlib`.
- No external network access from the sandbox.
- Has timeout and memory limits.
- Generated code should be reviewed for correctness before production use.

---

### 7.3 `web_search` configuration

```python
response = client.responses.create(
    model=model_deployment,
    instructions="Use web search when current information is required.",
    input="What are three major announcements from Microsoft Build this week?",
    tools=[
        {"type": "web_search"}
    ]
)
```

Key points:

- Useful for current, changing, or public information.
- Can reduce hallucination by grounding in current sources.
- Source quality varies.
- Results can change over time.
- Web retrieval can increase latency and token usage.

---

### 7.4 `file_search` configuration

```python
vector_store = client.vector_stores.create(name="policy-docs")

client.vector_stores.files.upload_and_poll(
    vector_store_id=vector_store.id,
    file=open("expenses_policy.pdf", "rb")
)

response = client.responses.create(
    model=model_deployment,
    instructions="Answer using the uploaded policy documents.",
    input="What is the maximum taxi claim amount?",
    tools=[
        {
            "type": "file_search",
            "vector_store_ids": [vector_store.id]
        }
    ],
    include=["file_search_call.results"]
)
```

Key points:

- Requires files to be uploaded and indexed into a vector store.
- Retrieves relevant chunks for grounding.
- Useful for internal policies, manuals, contracts and technical documents.
- Retrieval quality depends on document quality, chunking, scope and freshness.
- Updated files may need re-indexing.
- `include=["file_search_call.results"]` is useful for debugging and traceability.

---

### 7.5 `function` tool configuration

```python
function_tools = [
    {
        "type": "function",
        "name": "get_time",
        "description": "Get the current time"
    }
]

response = client.responses.create(
    model=model_deployment,
    input=messages,
    tools=function_tools
)
```

When the model returns a function call:

```python
for item in response.output:
    if item.type == "function_call" and item.name == "get_time":
        current_time = get_time()
        messages.append({
            "type": "function_call_output",
            "call_id": item.call_id,
            "output": current_time
        })

        response = client.responses.create(
            model=model_deployment,
            input=messages,
            tools=function_tools
        )
```

Key points:

- The model requests a function call.
- Your app validates and executes the function.
- Your app returns `function_call_output` with the `call_id`.
- The model uses the function result to produce the final answer.

## 8. Exam Traps

| Trap | Correct Understanding |
|---|---|
| Confusing Responses API tools with Foundry Tools. | Responses API tools are specified in model requests; Foundry Tools are Azure AI APIs such as Language, Speech and Document Intelligence. |
| Assuming a model can access current facts by default. | It needs `web_search` or another external data source for current information. |
| Assuming a model can answer from private documents automatically. | You need `file_search` with uploaded/indexed files in a vector store. |
| Thinking `file_search` is the same as web search. | `file_search` searches your indexed files; `web_search` searches public web content. |
| Thinking the model runs custom functions itself. | Your application runs the function and returns `function_call_output`. |
| Trusting function arguments without validation. | Always validate inputs and permissions before running functions. |
| Using function calling for high-impact actions without controls. | Use authorization, approval gates, logging and human review where needed. |
| Assuming `code_interpreter` can browse the web. | It runs Python in a sandbox with no external network access. |
| Assuming `code_interpreter` results are always correct. | Review generated code and results, especially in production or high-stakes use cases. |
| Assuming web results are always reliable. | Source quality varies and outputs require validation. |
| Using one huge vector store for unrelated domains. | Scope vector stores carefully to improve retrieval precision. |
| Forgetting to re-index updated files. | Updated content may not be searchable until re-uploaded/re-indexed. |
| Thinking tool use has no cost/performance impact. | Tool calls can increase latency, token usage and cost. |
| Using API keys casually in production. | Prefer Entra ID; store keys securely if required. |
| Forgetting cleanup after labs. | Delete resources to avoid Azure cost. |
