## 2. What This Module Teaches

This module teaches how to plan an AI solution on Azure before implementation.

Key idea:

> Azure AI solutions are not just model calls. They can combine models, AI services, prompt engineering, tools, agents, custom code, knowledge sources, endpoints, SDKs, monitoring, governance, and responsible AI controls.

Microsoft Foundry is introduced as the recommended platform for most Azure AI development because it provides project organisation, resource management, model access, agent development, tools, endpoints, portal experience, and SDK support.

## 3. Key Concepts

### AI Capability Areas

| Capability | What It Means | Best-Fit Use |
|---|---|---|
| **Generative AI and agents** | LLM-based apps that generate responses; agents add instructions and tools | Chat apps, task automation, AI assistants |
| **Natural language processing** | Text analysis using language models or statistical/task-specific NLP | Entity extraction, sentiment, summarisation |
| **Computer speech** | Speech recognition and speech synthesis | Voice apps, transcription, spoken responses |
| **Computer vision** | Image/video understanding and generation | Image analysis, visual QA, video analysis |
| **Information extraction** | Extract key facts from documents, forms, media, and content | Receipts, invoices, forms, document processing |

### Microsoft Foundry

Microsoft Foundry is the main Azure platform for building AI apps and agents.

It includes:

- **Foundry portal** — visual interface for projects, models, agents, tools, endpoints, and access
- **Foundry SDK** — programmatic access for automation, apps, scripts, and CI/CD
- **Foundry projects** — organise solution-specific assets
- **Foundry resources** — Azure resource layer that supports one or more projects
- **Models** — deployed model assets from Foundry Models
- **Agents** — LLM + instructions + tools
- **Tools** — built-in, custom, third-party, MCP, and Foundry Tools
- **Knowledge** — data sources used to ground agents and responses

### Foundry Resource vs Foundry Project

| Item | Meaning | Exam Tip |
|---|---|---|
| **Foundry resource** | Azure-level resource providing shared services, tools, compute, storage, access, and resource endpoint | Think shared infrastructure |
| **Foundry project** | Project-level container for models, agents, connections, knowledge, and project endpoint | Think solution workspace |
| **Default project** | One project under a resource can be designated as default | Useful for tooling/SDK workflows |

**Exam trap:**  
A Foundry resource and Foundry project are related but not the same.

## 5. Decision Rules

| Scenario | Choose | Why |
|---|---|---|
| Full Azure AI app or agent solution | **Microsoft Foundry project** | Best project/resource organisation |
| Very simple isolated AI service use case | Individual AI resource may be enough | Lower overhead |
| Need model-based chat app | **Foundry model deployment** | Gives endpoint for app integration |
| Need autonomous or semi-autonomous task execution | **Foundry Agent Service** | Agents combine model, instructions, and tools |
| Need external/custom tool access | **MCP or custom tools** | Standardised tool connectivity |
| Need shared knowledge grounding for agents | **Foundry IQ** | Central knowledge connection |
| Need entity/sentiment/summarisation | **Azure Language** | Purpose-built text analysis |
| Need speech input/output | **Azure Speech** | Purpose-built speech service |
| Need document field extraction | **Document Intelligence** | Better for forms, invoices, receipts |
| Need multimodal extraction | **Content Understanding** | Works across documents, images, audio, video |
| Need production authentication | **Microsoft Entra ID** | Better than key-only authentication |
| Need quick no-code testing | **Foundry portal playground** | Validate model/agent behaviour quickly |
| Need IDE-based development | **VS Code + Foundry Toolkit** | Developer productivity |

## 6. Architecture Pattern

### Foundry-Based AI Solution

```text
User / Client App
   ↓
Application Code
   ↓
Foundry Project Endpoint
   ↓
Model Deployment or Agent
   ↓
Tools / Foundry IQ / Knowledge / Foundry Tools
   ↓
Safety + Identity + Monitoring + Governance
   ↓
Response / Action
```

### Foundry Resource and Project Structure

```text
Azure Subscription
  ↓
Resource Group
  ↓
Microsoft Foundry Resource
  ├── Resource endpoint
  ├── Shared Foundry Tools
  ├── Access management
  └── Foundry Project(s)
        ├── Model deployments
        ├── Agents
        ├── Connections
        ├── Knowledge sources
        ├── Project endpoint
        └── Azure OpenAI endpoint
```

### Endpoint Decision

| Endpoint | Use For |
|---|---|
| **Project endpoint** | Foundry-specific APIs, Foundry Agent Service, Foundry models through project context |
| **Azure OpenAI endpoint** | OpenAI-compatible APIs such as Chat Completions and Responses API |
| **Tool-specific endpoint** | Foundry Tools such as Language, Speech, Translator, Document Intelligence, Content Understanding |

## 7. Configuration / Implementation Notes

### Foundry Project Setup

Remember the basic flow:

1. Open Microsoft Foundry portal.
2. Create a new project.
3. Select subscription, resource group, region, and Foundry resource.
4. Deploy a model from the model catalog.
5. Test the model in the playground.
6. Review resource-level and project-level details.
7. Note project endpoint, Azure OpenAI endpoint, and keys.
8. Install and connect VS Code Foundry Toolkit.
9. Clean up resources to avoid cost.

### Authentication

| Method | Use |
|---|---|
| **Key-based authentication** | Simple development/testing |
| **Microsoft Entra ID / token-based authentication** | Preferred for production |

### Developer Tools

| Tool | Why It Matters |
|---|---|
| **Foundry portal** | Manage and test AI assets visually |
| **VS Code Foundry Toolkit** | Work with models, agents, connections, vector stores in IDE |
| **GitHub** | Source control and DevOps |
| **GitHub Copilot** | Developer productivity |
| **Foundry SDK** | Automate and integrate Foundry assets |
| **OpenAI APIs/SDKs** | Build apps using compatible model APIs |
| **Foundry Tools SDKs / REST APIs** | Use service-specific tool capabilities |

## 8. Exam Traps

| Trap | Correct Understanding |
|---|---|
| Thinking Foundry is only a model playground | Foundry is a full AI development platform |
| Confusing Foundry resource with Foundry project | Resource = Azure shared infrastructure; Project = solution workspace |
| Assuming every task needs a generative AI agent | Foundry Tools can be cheaper, more predictable, and purpose-built |
| Using keys as the default production security model | Entra ID/token-based authentication is preferred for production |
| Confusing project endpoint and Azure OpenAI endpoint | They support different integration patterns |
| Ignoring responsible AI until deployment | Responsible AI must be considered from planning through operation |
| Thinking agents are just chatbots | Agents combine model, instructions, tools, knowledge, and task responsibility |
| Treating Document Intelligence and Content Understanding as identical | Document Intelligence is document/form extraction; Content Understanding is broader multimodal content analysis |
| Assuming classic Foundry hub architecture is the latest model | This module focuses on the newer Foundry project architecture |
