## 2. What This Module Teaches

This module teaches how to plan an AI solution on Azure before implementation.

Key idea:
> Azure AI solutions are not just model calls. They can combine models, AI services, prompt engineering, tools, agents, custom code, knowledge sources, endpoints, SDKs, monitoring, governance, and responsible AI controls.

Microsoft Foundry is introduced as the **recommended platform for most Azure AI development** because it provides:
- Project organisation
- Resource management
- Model access
- Agent development
- Tools integration
- Endpoints and SDKs
- Portal experience

## 3. Key Concepts

### 3.1 AI Capability Areas

| Capability | What It Means | Best-Fit Use | Example |
|---|---|---|---|
| **Generative AI and agents** | LLM-based apps that generate responses; agents add instructions and tools | Chat apps, task automation, AI assistants | Customer support chatbot |
| **Natural language processing** | Text analysis using language models or statistical/task-specific NLP | Entity extraction, sentiment, summarisation | Extract invoice data, analyze reviews |
| **Computer speech** | Speech recognition and speech synthesis | Voice apps, transcription, spoken responses | Real-time transcription, voice assistant |
| **Computer vision** | Image/video understanding and generation | Image analysis, visual QA, video analysis | Document scanning, facial recognition |
| **Information extraction** | Extract key facts from documents, forms, media, and content | Receipts, invoices, forms, document processing | Extract line items from receipts |

**Exam tip:** Know which capability to use for specific scenarios.

---

### 3.2 Microsoft Foundry Architecture

Microsoft Foundry is the main Azure platform for building AI apps and agents.

**Core Components:**

| Component | Purpose | What to Know |
|---|---|---|
| **Foundry portal** | Visual interface for projects, models, agents, tools, endpoints, and access | Web-based at https://ai.azure.com |
| **Foundry SDK** | Programmatic access for automation, apps, scripts, and CI/CD | Python: `azure-ai-projects` package |
| **Foundry projects** | Organise solution-specific assets | One default project per resource |
| **Foundry resources** | Azure resource layer that supports one or more projects | Shared infrastructure layer |
| **Models** | Deployed model assets from Foundry Models catalog | Over 1,900 models available |
| **Agents** | LLM + instructions + tools | Foundry Agent Service |
| **Tools** | Built-in, custom, third-party, MCP, and Foundry Tools | Available through project |
| **Knowledge** | Data sources used to ground agents and responses | Via Foundry IQ or vector stores |

---

### 3.3 Foundry Resource vs Foundry Project

| Item | Meaning | Scope | Exam Tip |
|---|---|---|---|
| **Foundry resource** | Azure-level resource providing shared services, tools, compute, storage, access, and resource endpoint | Shared across projects | Think shared infrastructure |
| **Foundry project** | Project-level container for models, agents, connections, knowledge, and project endpoint | Single project workspace | Think solution workspace |
| **Default project** | One project under a resource can be designated as default | Used by tooling/SDKs | Simplifies SDK workflows |

⚠️ **CRITICAL DISTINCTION:** A Foundry resource and Foundry project are related but NOT the same.

---

### 3.4 Foundry Tools (vs Tools in Responses API)

**Important:** Foundry Tools are Azure AI services, NOT the same as "tools" in the Responses API.

| Service / Tool | Role | Best For |
|---|---|---|
| **Azure Language** | Text analysis, entity extraction, sentiment, summarisation, question answering | Analyzing text documents and feedback |
| **Azure Speech** | Speech-to-text, text-to-speech, live voice for conversational apps | Voice input/output for applications |
| **Azure Translator** | State-of-the-art language translation | Multi-language support |
| **Azure Document Intelligence** | Extract fields from documents/forms (invoices, receipts, contracts) | Structured document extraction |
| **Azure Content Understanding** | Multimodal content analysis (documents, images, audio, video) | Broader content extraction scenarios |

**How to use:** Create client applications connecting to tool-specific endpoint in Foundry resource using APIs and SDKs.

## 5. Decision Rules

| Scenario | Choose | Why |
|---|---|---|
| Full Azure AI app or agent solution | **Microsoft Foundry project** | Best project/resource organisation |
| Very simple isolated AI service use case | Individual AI resource may be enough | Lower overhead |
| Need model-based chat app | **Foundry model deployment** | Gives endpoint for app integration |
| Need autonomous or semi-autonomous task execution | **Foundry Agent Service** | Agents combine model, instructions, and tools |
| Need external/custom tool access | **MCP or custom tools** | Standardised tool connectivity |
| Need shared knowledge grounding for agents | **Foundry IQ** | Central knowledge connection |
| Need entity/sentiment/summarisation | **Azure Language** | Purpose-built text analysis |
| Need speech input/output | **Azure Speech** | Purpose-built speech service |
| Need document field extraction | **Document Intelligence** | Better for forms, invoices, receipts |
| Need multimodal extraction | **Content Understanding** | Works across documents, images, audio, video |
| Need production authentication | **Microsoft Entra ID** | Better than key-only authentication |
| Need quick no-code testing | **Foundry portal playground** | Validate model/agent behaviour quickly |
| Need IDE-based development | **VS Code + Foundry Toolkit** | Developer productivity |

## 6. Responsible AI Principles

Microsoft's six core principles for responsible AI:

| Principle | Meaning | Example |
|---|---|---|
| **Fairness** | Treat all people fairly; avoid bias based on gender, ethnicity, etc. | Loan approval model treats all applicants equally |
| **Reliability and Safety** | Perform reliably and safely; minimize risk to human life | Autonomous vehicle AI has rigorous testing |
| **Privacy and Security** | Respect privacy; protect personal data and customer content | Encrypt data, implement access controls |
| **Inclusiveness** | Empower everyone; bring benefits to all parts of society | Design tested by diverse user groups |
| **Transparency** | Be understandable; communicate capabilities and limitations | Tell users how the system works and its limits |
| **Accountability** | People are responsible for AI systems; ensure governance and standards | Define accountability frameworks and monitoring |

**Exam tip:** System messages influence behavior but **don't guarantee** compliance. Always combine with testing, evaluation, and guardrails.

## 7. Configuration / Implementation Notes

### Foundry Project Setup

Basic flow:

1. Open Microsoft Foundry portal (https://ai.azure.com)
2. Create a new project with unique name
3. Select subscription, resource group, region
4. Foundry resource: Use default name (usually {project_name}-resource)
5. Deploy a model from the model catalog
6. Test the model in the playground
7. Review resource-level and project-level details
8. Note project endpoint, Azure OpenAI endpoint, and keys
9. Install and connect VS Code Foundry Toolkit
10. Clean up resources to avoid cost

### Authentication

| Method | Use | Best Practice |
|---|---|---|
| **Key-based authentication** | Simple development/testing | Development only |
| **Microsoft Entra ID / token-based authentication** | Preferred for production | Always use in production |

## 8. Exam Traps

| Trap | Correct Understanding |
|---|---|
| Thinking Foundry is only a model playground | Foundry is a full AI development platform |
| Confusing Foundry resource with Foundry project | Resource = Azure shared infrastructure; Project = solution workspace |
| Assuming every task needs a generative AI agent | Foundry Tools can be cheaper, more predictable, and purpose-built |
| Using keys as the default production security model | Entra ID/token-based authentication is preferred for production |
| Confusing project endpoint and Azure OpenAI endpoint | They support different integration patterns |
| Ignoring responsible AI until deployment | Responsible AI must be considered from planning through operation |
| Thinking agents are just chatbots | Agents combine model, instructions, tools, knowledge, and task responsibility |
| Treating Document Intelligence and Content Understanding as identical | Document Intelligence is document/form extraction; Content Understanding is broader multimodal |
| Assuming classic Foundry hub architecture is the latest model | This module focuses on the newer Foundry project architecture |
| Confusing Foundry Tools with tools in Responses API | Foundry Tools = Azure services; Responses API tools = code_interpreter, web_search, file_search, function |
