The next major topic in the AI Solution Architect path is:

# Evaluation & Observability

This is one of the most important production topics because it answers:

> "How do we know the AI system is actually working?"

---

# Why Evaluation Matters

Traditional software:

```text
2 + 2 = 4
```

Easy to test.

---

AI systems:

```text
Question:
What is our leave policy?

Answer:
Employees receive 20 days annual leave.
```

Is it:

* Correct?
* Relevant?
* Grounded?
* Complete?
* Up to date?

Much harder.

---

# The Four Questions Every Enterprise Asks

## 1. Did Retrieval Work?

```text
User Query
    ↓
Retriever
    ↓
Relevant Documents?
```

Example:

```text
Question:
What is APRA CPS 230?

Retrieved:
CPS 234 documents
```

Retrieval failed.

---

## 2. Was the Answer Grounded?

Did the answer come from retrieved documents?

```text
Retrieved Context
      ↓
LLM Answer
```

Or did the model make something up?

---

## 3. Was the Answer Correct?

Even grounded answers can be wrong.

Example:

Document says:

```text
20 days leave
```

LLM says:

```text
25 days leave
```

Grounded but incorrect.

---

## 4. Was the Answer Useful?

User asks:

```text
How many leave days do I get?
```

Answer:

```text
Refer to HR Policy 5.2.
```

Technically correct.

Practically useless.

---

# The Core Evaluation Metrics

These are the ones architects should know.

| Metric              | Question                             |
| ------------------- | ------------------------------------ |
| Retrieval Relevance | Did we retrieve the right documents? |
| Context Precision   | Was retrieved context useful?        |
| Context Recall      | Did we miss important context?       |
| Groundedness        | Is answer supported by context?      |
| Faithfulness        | Did the model invent facts?          |
| Answer Relevance    | Did it answer the question?          |
| Correctness         | Is it factually correct?             |
| Helpfulness         | Is it useful to the user?            |

---

# The Three Levels of Evaluation

## Level 1 — Retrieval Evaluation

Evaluate:

```text
Query
   ↓
Retrieval
```

Questions:

* Were the right chunks found?
* Was ranking correct?
* Was recall sufficient?

---

## Level 2 — Generation Evaluation

Evaluate:

```text
Context
    ↓
LLM
    ↓
Answer
```

Questions:

* Grounded?
* Faithful?
* Correct?

---

## Level 3 — End-to-End Evaluation

Evaluate:

```text
User
   ↓
System
   ↓
Answer
```

Questions:

* Did the user get what they needed?
* Would they ask again?
* Did task completion improve?

---

# Observability

Evaluation tells you quality.

Observability tells you what happened.

Think:

```text
Logs
Metrics
Tracing
Monitoring
```

## Groundedness vs Faithfulness vs Correctness vs Relevance

This is the most commonly confused area in AI evaluation and is where most architects initially struggle. Once you understand those four terms, the rest of evaluation becomes much easier.


| Pattern                      | Purpose                                                         |
| ---------------------------- | --------------------------------------------------------------- |
| Basic RAG                    | Retrieve → Generate                                             |
| Hybrid RAG                   | Combine vector + keyword search                                 |
| Router RAG                   | Route query to the correct knowledge source                     |
| Query Transformation RAG     | Improve retrieval through query rewriting/expansion             |
| Corrective RAG (CRAG)        | Evaluate retrieval quality and retry if needed                  |
| Self-RAG                     | Model evaluates retrieval needs and retrieval quality itself    |
| Parent-Child RAG             | Retrieve small chunks but provide larger context                |
| Multi-Hop RAG                | Retrieve information across multiple reasoning steps            |
| Graph RAG                    | Retrieve entities and relationships from a knowledge graph      |
| Agentic RAG                  | Agents plan retrieval and tool usage                            |
| Adaptive RAG                 | Dynamically choose retrieval strategy based on query complexity |
| Fusion RAG / Multi-Query RAG | Generate multiple queries and combine retrieval results         |
| Hierarchical RAG             | Retrieve coarse → fine-grained information in stages            |
| Multimodal RAG               | Retrieve text, images, audio, video, PDFs                       |
| Knowledge Graph + RAG        | Hybrid Graph and Vector retrieval                               |
| Federated RAG                | Retrieve across multiple repositories/systems                   |
| SQL RAG / Structured RAG     | Retrieve from databases and structured sources                  |
| Long-Context RAG             | Use large context windows with retrieval optimization           |



This is actually a very good handbook, and after reading it I can see why **Self-RAG was omitted**.

It wasn't an accident.

The handbook is structured around patterns that are:

1. **Architecturally distinct**
2. **Common in enterprise implementations**
3. **Represent separate retrieval architectures**

Self-RAG is a bit different.

---

# Why Self-RAG Doesn't Naturally Fit

Look at your progression:

```text
Parent-Child
    ↓
Multi-Hop
    ↓
Graph RAG
    ↓
Agentic RAG
```

Each introduces a new architecture:

| Pattern      | New Architectural Capability   |
| ------------ | ------------------------------ |
| Parent-Child | Parent-child storage structure |
| Multi-Hop    | Iterative retrieval workflow   |
| Graph RAG    | Knowledge graph                |
| Agentic RAG  | Planning and orchestration     |

Self-RAG doesn't introduce a new retrieval architecture.

Instead it adds:

```text
Self-Evaluation
Reflection
Retrieval Decision Making
```

inside the model.

---

# Where Self-RAG Fits Better

If I were updating your handbook I'd put it here:

```text
Retrieval Enhancement Patterns

Hybrid RAG
Router RAG
Query Transformation
Corrective RAG
Self-RAG
```

because Self-RAG is essentially an evolution of Corrective RAG.

---

# Compare CRAG vs Self-RAG

## CRAG

```text
Retrieve
    ↓
External Evaluator
    ↓
Retry
```

Question:

> Did retrieval work?

The evaluator is outside the main LLM.

---

## Self-RAG

```text
LLM
 ↓
Need Retrieval?
 ↓
Retrieve
 ↓
Useful?
 ↓
Retrieve Again?
 ↓
Answer
```

Question:

> Can the model evaluate retrieval itself?

---

# Why Enterprise Architects Rarely Draw It

When you draw:

```text
Agent
   ↓
Retrieve
   ↓
Reflect
   ↓
Retrieve Again
```

you've effectively implemented Self-RAG behaviour.

Modern systems often absorb Self-RAG into:

* Agent frameworks
* Reflection loops
* Planning loops
* Retrieval controllers

rather than exposing it as a standalone box.

---

# If We Updated Your Handbook

I would revise the Retrieval Enhancement section to:

| Pattern              | Purpose                                                                                |
| -------------------- | -------------------------------------------------------------------------------------- |
| Hybrid RAG           | Improve retrieval quality                                                              |
| Router RAG           | Choose retrieval source                                                                |
| Query Transformation | Improve query quality                                                                  |
| Corrective RAG       | Validate retrieval quality                                                             |
| Self-RAG             | Allow the model to decide when retrieval is needed and whether retrieval is sufficient |

# Groundedness vs Faithfulness vs Correctness vs Relevance

This is probably the most misunderstood area in AI evaluation.

Most people use these terms interchangeably.

They are not the same.

---

# The Example

Suppose the retrieved document says:

```text
Employees receive 20 days annual leave.
```

User asks:

```text
How many annual leave days do employees receive?
```

---

# 1. Groundedness

Question:

> Is the answer supported by the retrieved context?

Retrieved Context:

```text
Employees receive 20 days annual leave.
```

Answer:

```text
Employees receive 20 days annual leave.
```

Grounded?

✅ Yes

The answer is directly supported by the context.

---

Answer:

```text
Employees receive 25 days annual leave.
```

Grounded?

❌ No

The context never said 25.

---

# Mental Model

```text
Groundedness
=
Answer supported by retrieved evidence?
```

---

# 2. Faithfulness

Question:

> Did the model faithfully use the information provided?

Context:

```text
Employees receive 20 days annual leave.
```

Answer:

```text
Employees receive 20 days annual leave.
```

Faithful?

✅ Yes

---

Answer:

```text
Employees receive 20 days annual leave and 10 wellness days.
```

Faithful?

❌ No

The model added information.

---

# Mental Model

```text
Faithfulness
=
No hallucination
```

---

# Groundedness vs Faithfulness

These are very close.

Think:

```text
Groundedness
=
Supported by evidence

Faithfulness
=
```

# Architect Takeaway

Think of evaluation in layers:

```text
Relevance
    ↓
Groundedness
    ↓
Faithfulness
    ↓
Correctness
```

And remember:

> A RAG system can be perfectly grounded and faithful while still being factually incorrect if the retrieved knowledge is outdated, incomplete, or wrong.

That single sentence explains why enterprises invest so much effort in knowledge management, governance, and observability—not just better models.

### One-line definitions

* **Groundedness** → Is the answer supported by retrieved evidence?
* **Faithfulness** → Did the model avoid inventing facts?
* **Correctness** → Is the answer objectively true?
* **Relevance** → Did the answer address the user's question?

This forms the foundation for understanding evaluation frameworks such as RAGAS, which is usually the next concept architects encounter.



| Metric           | What It Measures                                         | Key Question                                           | Depends On Retrieved Context? | Can Be True While Correctness Is False? | Example Failure                                                              |
| ---------------- | -------------------------------------------------------- | ------------------------------------------------------ | ----------------------------- | --------------------------------------- | ---------------------------------------------------------------------------- |
| **Groundedness** | Whether the answer is supported by retrieved evidence    | **"Is the answer backed by the retrieved documents?"** | Yes                           | Yes                                     | Document says 20 days leave, answer says 25 days                             |
| **Faithfulness** | Whether the model avoided adding unsupported information | **"Did the model stick to the evidence?"**             | Yes                           | Yes                                     | Document says 20 days leave, answer says 20 days + 10 wellness days          |
| **Correctness**  | Whether the answer is objectively true in the real world | **"Is the answer actually correct?"**                  | No                            | N/A                                     | Retrieved document is outdated (20 days), actual policy is 25 days           |
| **Relevance**    | Whether the answer addresses the user's question         | **"Did we answer what was asked?"**                    | No                            | Yes                                     | Answer says "Refer to HR Policy 5.2" instead of giving the leave entitlement |

# Why We Need These Metrics

Suppose the user asks:

> "What are the requirements of APRA CPS 230?"

Your retriever searches 10,000 documents.

The question becomes:

> Did we retrieve the right documents?

That's where Precision and Recall come from.

---

# Traditional Precision

Question:

> Of the documents we retrieved, how many were actually useful?

Example:

Relevant documents:

```text
Doc A
Doc B
Doc C
Doc D
```

Retriever returns:

```text
Doc A
Doc B
Doc X
Doc Y
```

---

Analysis:

```text
Retrieved = 4

Relevant = 2
```

Precision:

Precision = \frac{Relevant\ Retrieved}{Total\ Retrieved}

Result:

```text
2 / 4 = 50%
```

---

# Mental Model

```text
Precision
=
How clean are the results?
```

High precision:

```text
Retrieved mostly useful documents
```

---

# Traditional Recall

Question:

> Did we retrieve everything important?

Relevant documents:

```text
Doc A
Doc B
Doc C
Doc D
```

Retriever returns:

```text
Doc A
Doc B
```

Recall:

Recall = \frac{Relevant\ Retrieved}{Total\ Relevant}

Result:

```text
2 / 4 = 50%
```

---

# Mental Model

```text
Recall
=
How much important information did we find?
```

---

# Precision vs Recall

Imagine fishing.

### High Precision

```text
10 fish caught
9 fish wanted
```

Very clean.

---

### High Recall

```text
Found almost every fish
but caught lots of junk too
```

Complete but messy.

* Context Recall

```text

```

```text
2 / 4 = 50%
```

# Evaluation & Observability Handbook

# Why Evaluation Matters

Traditional software can usually be tested using deterministic assertions.

```text
Input
  ↓
Expected Output
  ↓
Pass / Fail
```

AI systems are probabilistic.

```text
Question
   ↓
Retrieval
   ↓
LLM
   ↓
Answer
```

Multiple answers may be acceptable.

Evaluation therefore becomes a measurement problem rather than a pass/fail problem.

---

# Evaluation Hierarchy

```text
Evaluation
│
├── Retrieval Evaluation
│
├── Generation Evaluation
│
├── Operational Evaluation
│
└── Business Evaluation
```

# Retrieval Evaluation

Purpose:

Determine whether the retrieval system found the right information.

---

## Precision

### Question

How many retrieved items were actually useful?

### Formula

```text
Relevant Retrieved
------------------
Total Retrieved
```

### Mental Model

```text
Avoid Junk
```

### High Precision

Most retrieved content is relevant.

### Low Precision

Many irrelevant results are retrieved.

---

## Recall

### Question

How many useful items were found?

### Formula

```text
Relevant Retrieved
------------------
Total Relevant
```

### Mental Model

```text
Avoid Missing Information
```

### High Recall

Most relevant information is retrieved.

### Low Recall

Important information is missed.

---

## Context Precision

### Question

How much retrieved context was useful for answering the question?

### Mental Model

```text
How much irrelevant context was retrieved?
```

### Example

Retrieved:

* CPS 230
* CPS 234
* Holiday Policy
* Cafeteria Menu

Only CPS 230 is useful.

Context Precision is poor.

---

## Context Recall

### Question

Did retrieval capture all context needed to answer the question?

### Mental Model

```text
Did we retrieve enough information?
```

### Example

Required:

* Requirement A
* Requirement B
* Requirement C
* Requirement D

Retrieved:

* Requirement A
* Requirement B

Context Recall is poor.

---

| -------------------- | ------------------ |
| Hybrid RAG           | Precision + Recall |
| Query Transformation | Recall             |
| Re-ranking           | Precision          |
| Parent-Child         | Context Recall     |
| Router RAG           | Precision          |
| Graph RAG            | Context Recall     |
| Multi-Hop            | Recall             |
| Corrective RAG       | Precision + Recall |

# Generation Evaluation

Purpose:

Determine whether the generated answer is good.

---

## Groundedness

### Question

Is the answer supported by retrieved evidence?

### Mental Model

```text
Supported by evidence?
```

### Example

Context:

```text
Employees receive 20 days annual leave.
```

Answer:

```text
Employees receive 20 days annual leave.
```

Grounded = Yes

---

## Faithfulness

### Question

Did the model faithfully use the retrieved evidence?

### Mental Model

```text
No hallucinations?
```

### Example

Context:

```text
20 days leave
```

Answer:

```text
20 days leave plus wellness leave
```

Faithfulness = No

---

## Correctness

### Question

Is the answer objectively true?

### Mental Model

```text
Real-world truth
```

### Important Insight

A response may be:

* Grounded
* Faithful

and still be incorrect if the source information is outdated.

---

## Relevance

### Question

Did the answer address the user's question?

### Mental Model

```text
Did we answer the question?
```

### Example

Question:

```text
How many leave days?
```

Answer:

```text
Refer to HR Policy Section 5.2
```

Relevant = Poor

---

# Comparison Table

| Metric       | Measures                | Key Question                     |
| ------------ | ----------------------- | -------------------------------- |
| Groundedness | Evidence support        | Is answer supported by context?  |
| Faithfulness | Hallucination avoidance | Did model stay within evidence?  |
| Correctness  | Truth                   | Is answer objectively true?      |
| Relevance    | User value              | Did answer address the question? |

---

# Important Distinction

A response can be:

* Grounded
* Faithful

and still be:

* Incorrect

if the retrieved information itself is wrong or outdated.

# Query Transformation

## Purpose

Improve retrieval by improving the query.

---

## Problem

Users ask vague questions.

```text
leave policy?
```

instead of:

```text
annual leave entitlement policy
```

## Retrieval Flow

```text

     ↓
Query Transformation
     ↓
Optimized Query
     ↓
Retrieval
```

## Techniques

### Query Rewriting

```text
Short → Precise
```

Example:

```text
leave policy
```

↓

```text
annual leave entitlement policy
```

---

### Query Expansion

```text
AI
```

↓

```text
Artificial Intelligence
Generative AI
Machine Learning
```

---

### Query Decomposition

Break complex questions into smaller questions.

---

### Multi-Query Generation

Generate multiple retrieval variants.

---

### HyDE

Hypothetical Document Embeddings.

Generate a hypothetical answer first.

Use that answer to perform retrieval.

---

### Step-Back Prompting

Move from specific question to broader concept.

---

## Key Insight

Better questions produce better retrieval.

Improve retrieval precision.
